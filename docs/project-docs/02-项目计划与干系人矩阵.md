# 项目计划与干系人矩阵

## 1. 文档目的

本文档用于说明 HR Agent 企业试点项目的组织分工、里程碑、评审节点、交付物和风险跟踪方式，支撑项目从立项到上线验收的全过程管理。

## 2. 项目组织结构

| 角色 | 职责 | 典型人员 |
| --- | --- | --- |
| 项目 Sponsor | 提供资源和决策支持，确认项目目标 | HR 负责人/数字化负责人 |
| 业务 Owner | 定义业务目标、验收业务流程 | HR 招聘负责人 |
| 产品负责人 | 需求拆解、范围控制、版本规划 | 产品经理 |
| 技术负责人 | 技术方案、架构评审、开发质量 | 后端/全栈负责人 |
| 前端开发 | 页面、交互、路由、API 对接 | 前端工程师 |
| 后端开发 | API、服务、数据库、AI 集成 | 后端工程师 |
| 测试负责人 | 测试计划、用例、缺陷跟踪、回归 | QA |
| 运维负责人 | 部署、配置、备份、日志、回滚 | 运维/平台工程师 |
| UAT 代表 | 代表真实用户验收系统 | HR 专员/培训管理员 |
| 安全/合规代表 | 简历、权限、数据安全审核 | 信息安全/法务 |

## 4. 里程碑计划

| 阶段 | 目标 | 主要活动 | 交付物 | 准出标准 |
| --- | --- | --- | --- | --- |
| M0 立项 | 明确为什么做、做什么、不做什么 | 背景调研、价值评估、范围确认 | 项目背景与目标说明 | Sponsor 和业务 Owner 确认 |
| M1 需求 | 明确业务流程和功能边界 | 用户访谈、流程梳理、PRD 评审 | 需求文档、权限矩阵、数据字典 | P0/P1 范围冻结 |
| M2 方案 | 明确系统如何实现 | 架构设计、接口设计、数据设计、AI 工作流设计 | 方案设计、API 清单、AI 质量说明 | 技术负责人确认可实施 |
| M3 开发 | 完成核心功能 | 前后端开发、数据库、Dify/LLM 接入 | 可运行版本 | 核心路径可联调 |
| M4 SIT | 验证系统功能和集成 | 功能测试、接口测试、异常测试 | 测试报告、缺陷列表 | 无 Blocker/Critical |
| M5 UAT | 业务验收 | HR、培训、管理员按场景验收 | UAT 验收单 | 业务方签字 |
| M6 上线 | 发布试点版本 | 部署、配置、备份、冒烟测试 | 上线检查表、回滚方案 | 上线评审通过 |

## 8. 变更管理

需求变更需记录：

- 变更编号
- 提出人
- 变更内容
- 影响模块
- 影响工期
- 是否影响上线范围
- 评审结论
- 责任人

变更原则：

- P0 范围冻结后，新增需求默认进入下一版本。
- 影响安全、权限、数据结构或上线时间的变更必须评审。
- 只修复缺陷不算需求变更，但如果改变业务规则，需要重新确认。
