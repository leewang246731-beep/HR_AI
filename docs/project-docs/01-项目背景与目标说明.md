# 项目背景与目标说明

## 1. 项目概述

HR Agent 是一个 AI 驱动的人力资源管理系统，面向企业 HR 招聘、培训和知识问答场景。系统以 Vue 3 前端、FastAPI 后端、PostgreSQL + pgvector 数据库、Dify 工作流、LLM 与 RAG 检索增强能力为基础，提供从自然语言需求到 HR 业务工具执行的统一入口。

项目的核心不是替代 HR 的业务判断，而是把重复、耗时、依赖文本处理的工作流程自动化、结构化和可追踪化。用户可以通过 HR Agent 对话入口表达需求，也可以进入具体功能页面完成 JD 生成、简历筛选、面试方案、试卷生成、知识问答和系统管理等任务。

## 2. 项目背景

企业 HR 团队在招聘和培训工作中长期面临以下问题：

- 招聘需求从业务部门口头描述到正式 JD，往往需要 HR 多轮整理和补写。
- 简历筛选依赖人工阅读，效率低，标准不稳定，候选人对比成本高。
- 面试方案、考察维度和评分标准容易因岗位、面试官经验不同而不一致。
- 培训考试出题、试卷管理、结果统计存在大量重复劳动。
- 企业制度、岗位资料、业务知识分散在文档中，HR 与员工检索成本高。
- 多个系统页面和流程割裂，新用户需要理解菜单、字段和操作顺序。

在大模型能力可用后，企业可以把文本理解、内容生成、文档检索、候选人匹配、知识问答等任务沉淀为标准化 AI 工作流。HR Agent 项目正是围绕这些高频、低结构化、可被 AI 增强的 HR 工作场景搭建。

### 2.1 立项动因

本项目适合由企业 HR 部门、数字化部门或业务中台团队共同发起。立项动因包括：

- 招聘需求增长，但 HR 团队人力有限，需要提升初筛和文档产出效率。
- 企业希望把大模型能力落在明确业务流程中，而不是停留在通用聊天工具层面。
- 招聘、培训、知识问答均属于企业内部高频文本处理场景，具备 AI 试点价值。
- 项目可从内部试点开始，不需要一次性改造完整 HRIS 或 ATS，落地风险相对可控。

### 2.2 当前工作方式与问题量化口径

正式立项时建议收集以下基线数据，用于评估项目价值：

| 指标 | 采集方式 | 用途 |
| --- | --- | --- |
| 每月新增招聘岗位数 | HR 招聘台账 | 评估 JD 生成和评分标准生成价值 |
| 每个岗位平均简历数 | 简历收集记录 | 评估简历筛选自动化价值 |
| HR 编写一份 JD 平均耗时 | 抽样访谈或工时记录 | 估算节省时间 |
| HR 初筛一份简历平均耗时 | 抽样计时 | 估算简历筛选效率提升 |
| 每月培训考试次数 | 培训计划 | 评估试卷生成和考试管理价值 |
| 企业制度/知识问答次数 | HR/行政问答记录 | 评估知识助手价值 |

如缺少历史数据，可在试点期记录 2-4 周作为项目基线。

## 3. 要解决的问题

### 3.1 业务问题

- 降低 HR 从招聘需求到可发布 JD 的产出成本。
- 提高简历筛选效率，并让筛选理由、评分维度更透明。
- 将面试方案从经验驱动改为基于 JD、简历、评分结果的结构化生成。
- 支持企业内部培训资料快速转化为考试内容。
- 将企业知识文档纳入知识库问答，减少人工查询和重复答疑。

### 3.2 产品问题

- 将 HR 的多个 AI 工具集中到一个系统内，而不是分散在独立脚本或外部工具中。
- 提供统一登录、权限、历史记录和数据管理能力。
- 通过对话入口降低使用门槛，让用户可以先说目标，再由系统推荐或调度工具。
- 对关键业务动作保留人工确认和可编辑环节，避免 AI 直接产生不可控操作。

### 3.3 技术问题

- 将 LLM、Dify 工作流、RAG 检索、文件解析、向量存储和业务数据库整合成可运行应用。
- 支持普通 REST 请求和 SSE 流式输出，改善长文本生成和多步骤任务体验。
- 通过 JWT、角色、超级管理员等机制保护系统管理和敏感配置。
- 将文档、对话、简历评估、试卷、邮箱配置等业务数据结构化存储。

## 4. 目标用户

### 4.1 主要用户

- HR 负责人：关注招聘效率、流程规范、候选人质量、团队协同与数据可视化。
- HR 专员：日常使用 JD 生成、简历筛选、面试方案、邮件通知、考试管理等功能。
- 培训管理员：上传学习资料，生成试卷，管理考试和结果。
- 系统管理员：负责用户、角色、邮箱配置、基础运行配置和权限维护。

### 4.2 次要用户

- 面试官：查看候选人分析、面试方案、评分维度和建议问题。
- 普通员工或业务部门用户：通过知识助手查询企业制度、培训材料或岗位相关知识。
- 候选人或考试参与者：通过考试分享链接参与考试，不一定需要系统账号。

## 5. 项目目标

### 5.1 业务目标

- 建立企业 HR 智能化工作台，覆盖招聘、培训、知识问答和系统管理核心场景。
- 将 JD 生成、评分标准、简历筛选、面试方案、试卷生成从人工撰写转为 AI 辅助生成。
- 提升 HR 工作效率，使 HR 能把更多时间投入沟通、判断和组织协同。
- 通过结构化数据留存招聘与培训过程，为后续复盘和优化提供基础。

### 5.2 产品目标

- 提供清晰的功能导航：HR Agent、智能招聘、智能培训、知识助理、系统管理。
- 提供自然语言 Agent 入口：支持用户输入任务，系统识别意图、判断前置条件、给出工具入口或执行结果。
- 保留功能页面入口：对于复杂表单、批量上传、历史管理等任务，用户可以直接进入业务页面操作。
- 提供可追踪结果：生成的 JD、评分标准、简历评估、面试方案、试卷和考试结果应可查询、管理和删除。

### 5.3 技术目标

- 后端采用 FastAPI 分层架构，按 endpoints、schemas、models、services、core、utils 组织。
- 前端采用 Vue 3 + Vite + Element Plus，实现菜单、鉴权、API 请求封装与业务页面。
- 数据层使用 PostgreSQL 存储业务数据，并通过 pgvector 支持知识库向量检索。
- AI 能力通过 LLM 服务、Dify 工作流、Embedding、Rerank 和 RAG 服务组合实现。
- 支持 JWT 登录认证、用户角色、超级管理员权限、异常处理、CORS 和健康检查。

## 6. 业务价值

### 6.1 效率价值

- 减少 JD、评分标准、面试题、试卷等文本材料的初稿撰写时间。
- 批量处理简历，减少 HR 初筛工作量。
- 将文档检索转化为问答，提高制度和知识查询效率。

### 6.2 质量价值

- 让 JD、评分标准、面试方案和试卷生成具备相对统一的结构。
- 将候选人评价从纯主观判断转为“评分 + 理由 + 维度”的辅助判断。
- 通过 RAG 引用企业文档内容，降低知识问答脱离资料的风险。

### 6.3 管理价值

- 用户、角色、文档、对话、评估结果和考试结果进入统一系统管理。
- 支持统计看板和活动记录，为管理者提供运营概览。
- 通过权限控制保护系统管理、邮箱配置等敏感操作。

### 6.4 体验价值

- HR Agent 对话入口降低新用户理解菜单和流程的成本。
- 流式生成让用户感知任务进度，避免长时间等待无反馈。
- 业务页面保留可编辑、可确认、可导出的操作空间。

### 6.5 投入产出分析口径

本项目 ROI 不应只用“AI 是否生成内容”衡量，而应从流程节省和质量提升衡量：

| 价值项 | 计算口径 |
| --- | --- |
| JD 生成节省时间 | 岗位数量 x 人工 JD 初稿耗时 - AI 辅助后编辑耗时 |
| 简历初筛节省时间 | 简历数量 x 人工初筛耗时 - AI 批量评分和复核耗时 |
| 面试准备节省时间 | 面试人数 x 人工准备方案耗时 - AI 方案编辑耗时 |
| 试卷生成节省时间 | 考试次数 x 人工出题耗时 - AI 生成和审核耗时 |
| 知识问答节省时间 | 问答次数 x 人工查询/答复耗时 - 自助问答耗时 |
| 质量提升 | JD 结构一致性、评分标准覆盖率、候选人评价可解释性、考试题覆盖率 |

### 6.6 组织价值

- 对 HR 团队：降低重复劳动，提高候选人分析和业务沟通时间占比。
- 对业务部门：更快获得岗位 JD 和候选人分析结果，缩短招聘启动周期。
- 对培训团队：把培训资料快速转为考试题，提高培训闭环效率。
- 对 IT/数字化团队：形成可复用的大模型业务应用架构，包括 RAG、工作流、权限、流式输出和文件处理能力。

## 7. 成功标准

### 7.1 功能成功标准

- 用户可以注册、登录，并根据权限访问对应功能。
- HR 可以完成 JD 生成、保存、查询、更新和删除。
- 系统可以根据 JD 或岗位需求生成评分标准。
- HR 可以上传简历并获得 AI 评分、匹配度分析和评分理由。
- HR 可以基于候选人简历和 JD 生成面试方案。
- 用户可以上传文档，建立知识库，并通过知识助手问答。
- 培训管理员可以基于文档生成试卷、发布分享链接、查看考试结果。
- HR Agent 可以识别常见 HR 任务意图，返回工具执行结果、追问缺失信息或引导用户到对应页面。

### 7.2 质量成功标准

- 核心接口在正常输入下返回成功结果，在缺失参数、无权限、资源不存在等场景下返回明确错误。
- 长文本生成和 Agent 执行支持流式反馈，前端不会出现长时间无响应。
- 敏感管理接口必须经过认证和权限校验。
- 上传文件大小、格式、解析失败等异常应有可理解提示。
- 数据库初始化、服务启动、健康检查和基础页面访问可稳定通过。

### 7.3 业务验收标准

- 选取至少 3 类典型岗位完成端到端招聘流程验证：需求输入、JD 生成、评分标准、简历评分、面试方案。
- 选取至少 2 份企业资料完成知识库上传、处理和问答验证。
- 选取至少 1 份培训资料完成试卷生成、考试分享、答题提交和结果查看。
- 管理员完成用户创建、角色分配、邮箱配置测试。
- 业务负责人确认 AI 结果可作为 HR 初稿或辅助判断使用，而不是不可解释的黑盒输出。

### 7.4 量化验收建议

| 维度 | 建议标准 |
| --- | --- |
| JD 生成效率 | 典型岗位 JD 初稿生成时间小于 3 分钟，人工编辑后可发布 |
| 简历评分效率 | 单份简历从上传到返回评分结果的等待时间满足试点可接受范围 |
| 面试方案质量 | HR 或面试官确认方案结构完整，问题与 JD/简历相关 |
| 知识问答质量 | 对测试文档中的明确事实问题，回答应与资料一致 |
| 试卷生成质量 | 题目覆盖上传资料核心内容，答案和解析可人工审核 |
| 权限安全 | 普通用户不能访问用户管理、角色管理和邮箱配置 |
| 可恢复性 | 上线前具备数据库和上传目录备份，支持回滚 |

### 7.5 决策门槛

项目进入试点上线前，应至少满足：

- P0 功能全部可用。
- 招聘、知识库、考试三条主流程至少各完成一次端到端验证。
- 敏感配置已替换默认值。
- 业务方确认 AI 结果定位为辅助产物，不直接作为最终人事决策。
- 运维方确认数据库、上传目录、日志和回滚方案已准备。

## 8. 项目边界

本项目当前定位为企业 HR AI 辅助工作台和智能化流程原型，不覆盖以下范围：

- 不作为完整 ATS 替代品，不覆盖招聘渠道投放、候选人 CRM、Offer 审批、入职全流程。
- 不作为完整 HRIS 替代品，不覆盖薪酬、绩效、考勤、组织架构全生命周期。
- 不承诺 AI 自动做最终录用、淘汰或绩效判断，最终决策应由业务人员负责。
- 不默认内建企业级 CI/CD、集中日志平台、服务发现、灰度发布平台，这些属于上线环境和平台协作范围。

## 9. 项目约束与假设

### 9.1 业务假设

- HR 用户愿意接受“AI 初稿 + 人工审核”的协作方式。
- 企业允许将内部文档、简历等资料接入受控 AI 服务。
- 试点阶段以内部 HR、培训和知识问答场景为主，不直接开放给外部候选人大规模使用。

### 9.2 技术约束

- LLM、Embedding、Rerank、Dify 等外部服务可用性会影响生成和检索体验。
- 上传文件解析质量受文件格式、扫描件、乱码、表格复杂度等因素影响。
- 当前项目具备应用级日志和异常处理，但企业级监控、告警、链路追踪需由上线环境补齐。
- 默认上传目录需要持久化和备份，否则重启或迁移可能造成文件丢失。

### 9.3 合规约束

- 简历属于个人敏感信息，上线前应明确访问权限、保存期限、删除流程和外部模型调用范围。
- AI 评分结果应作为辅助参考，避免被解释为自动化人事决策。
- 邮件发送、候选人通知等对外动作应保留人工确认。
