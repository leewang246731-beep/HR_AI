# 测试与验收文档

## 1. 文档目的

本文档说明如何验证 HR Agent 项目是否符合需求，包括测试范围、测试用例、测试数据、缺陷记录、验收标准、上线条件、验收角色、上线步骤和回滚方案。

## 2. 测试目标

- 验证核心 HR 招聘流程可以闭环。
- 验证培训试卷生成、考试分享、答题提交和结果查看可用。
- 验证知识库上传、处理、检索问答可用。
- 验证 HR Agent 能正确识别常见任务、追问缺失信息、调度工具或引导页面。
- 验证认证、授权、异常处理、文件上传限制和敏感配置保护有效。
- 验证上线前服务、数据库、外部 AI 工作流、前端构建和基础运维条件满足要求。

## 2.1 测试策略

测试采用分层验证：

- 单模块验证：逐个验证登录、JD、简历、面试、知识库、试卷、邮箱等模块。
- 端到端流程验证：覆盖招聘主流程、知识库问答流程、培训考试流程。
- 权限验证：以不同角色登录，验证菜单、接口和操作边界。
- AI 质量验证：用固定测试数据比较生成内容是否完整、稳定、可解释。
- 上线冒烟验证：上线前后执行最小关键路径检查。

## 2.2 测试轮次

| 轮次 | 目标 | 执行人 | 准出标准 |
| --- | --- | --- | --- |
| SIT-1 | 模块功能测试 | 测试/研发 | P0 功能主路径通过 |
| SIT-2 | 集成和异常测试 | 测试/研发 | 关键异常可处理，无 Critical 缺陷 |
| UAT | 业务验收 | HR/培训/管理员 | 业务主流程可接受 |
| Release Smoke | 上线冒烟 | 测试/运维/研发 | 上线检查全部通过 |

## 3. 测试范围

### 3.1 范围内

- 前端页面可访问性和主要交互。
- 后端 API 正常路径和异常路径。
- 登录、Token、角色和权限。
- JD、评分标准、简历评估、面试方案。
- 文档上传、知识库管理、知识问答。
- 试卷生成、试卷管理、考试分享、考试结果。
- 邮箱配置、连接测试、抓取日志。
- Agent 普通对话、流式对话、工具编排。
- 系统启动、健康检查、数据库连接、外部服务配置。

### 3.2 范围外

- 大规模压测和容量评估。
- 完整安全渗透测试。
- 企业级 CI/CD 流水线验证。
- 多租户隔离和商业化计费。
- 第三方招聘渠道真实投放。

范围外项目不代表不重要，而是需要在正式生产化阶段另立专项。

## 4. 测试环境

| 项目 | 要求 |
| --- | --- |
| 操作系统 | macOS/Linux/企业服务器环境均可 |
| 后端 | Python 3.10+，FastAPI |
| 前端 | Node.js，npm，Vue 3/Vite |
| 数据库 | PostgreSQL 13+，启用 pgvector |
| 外部服务 | Dify、LLM、Embedding、Rerank 可访问 |
| 浏览器 | Chrome、Edge、Safari 或 Firefox 现代版本 |

### 4.1 环境准入

- 后端服务可启动。
- 数据库可连接，pgvector 可用。
- 初始角色和超级管理员已创建。
- 前端可访问并能代理到后端。
- Dify、LLM、Embedding、Rerank 至少在测试环境有可用配置。
- 上传目录可写。

基础启动检查：

```bash
cd backend
docker compose up -d
python scripts/seed_roles.py
python main.py
```

```bash
cd frontend
npm install
npm run dev
```

健康检查：

```bash
curl http://localhost:8000/api/v1/health
```

预期返回：

```json
{
  "status": "healthy",
  "message": "HR Agent API正在运行"
}
```

### 5.2 招聘测试数据

岗位样例：

- 前端工程师：Vue 3、TypeScript、Element Plus、3 年经验、本科、上海。
- 后端工程师：Python、FastAPI、PostgreSQL、RAG、3-5 年经验、北京。
- AI 产品经理：大模型应用、需求分析、项目管理、B2B 软件经验、深圳。

简历样例：

- 高匹配候选人：技能、项目经验与岗位高度一致。
- 中匹配候选人：部分技能匹配，经验年限略低。
- 低匹配候选人：行业和技能差异明显。

### 5.3 文档测试数据

- 企业 HR 制度文档。
- 岗位说明书或培训材料。
- PDF、DOCX、TXT 各至少 1 份。
- 空文档或无法解析文档 1 份，用于异常测试。

### 5.4 考试测试数据

- 参考文档 1 份。
- 试卷配置：单选 5 题、多选 3 题、判断 2 题、难度中等、满分 100。
- 答题人 2 名：一名高分、一名低分。

严重级别定义：

- Blocker：系统无法启动、无法登录、核心流程完全不可用。
- Critical：数据错误、权限绕过、核心流程关键步骤失败。
- Major：主要功能异常但有绕行方案。
- Minor：文案、样式、非关键交互问题。

## 8. 验收标准

### 8.1 功能验收

- P0 用例全部通过。
- P1 用例通过率不低于 95%，且无未解决 Critical 缺陷。
- 招聘主流程至少覆盖 3 个岗位样例并完成闭环。
- 知识库至少完成 2 份文档上传、处理和问答。
- 试卷流程至少完成 1 次生成、分享、答题和结果查看。
- Agent 对常见任务不误触发、不漏追问，结果能引导到正确页面或工具。

### 8.2 安全验收

- 未登录访问主业务接口返回 401。
- 普通用户访问管理员接口返回 403。
- 生产环境密钥、数据库密码、LLM/Dify Key 不使用默认值。
- 上传文件大小和格式限制有效。
- 邮箱配置仅超级管理员可访问。

### 8.3 体验验收

- 前端菜单、页面标题、表单字段符合业务理解。
- 长任务有流式输出或进度反馈。
- 常见错误有明确提示。
- 生成结果可编辑、可保存、可查询。

### 8.4 技术验收

- 后端服务启动成功，健康检查通过。
- 前端构建成功。
- 数据库初始化和基础种子数据可用。
- 外部 Dify、LLM、Embedding、Rerank 服务连通。
- 关键错误会记录日志。

## 8.5 准入与准出标准

测试准入：

- 需求范围已确认。
- 测试环境可用。
- P0 功能代码已部署。
- 测试账号和测试数据已准备。
- 外部 AI 服务配置可用。

测试准出：

- P0 用例 100% 通过。
- 无未关闭 Blocker 和 Critical 缺陷。
- Major 缺陷有明确绕行方案和修复计划。
- 端到端测试场景通过。
- UAT 负责人完成签字确认。

## 9. 上线前条件

### 9.1 代码与构建

- 后端依赖安装成功。
- 前端 `npm run build` 成功。
- 必要数据库表和初始角色已初始化。
- 无 Blocker 和 Critical 缺陷。
- 已确认当前代码分支和版本号。

### 9.2 配置与密钥

- 替换生产 `SECRET_KEY`。
- 配置生产数据库连接。
- 配置 Dify、LLM、Embedding、Rerank Key。
- 配置 CORS 允许的正式前端域名。
- 配置上传目录和持久化存储。
- 邮箱配置通过连接测试。

### 9.3 数据与权限

- 创建超级管理员账号。
- 创建 HR、培训管理员、普通用户等基础角色。
- 确认管理员接口权限。
- 明确简历、考试结果、文档的保留和删除策略。

### 9.4 运维与支持

- 明确前端静态资源部署位置。
- 明确后端进程管理方式。
- 明确数据库备份方案。
- 明确日志保存位置。
- 明确外部 AI 服务不可用时的处理方式。
- 明确业务支持联系人和技术支持联系人。

### 9.5 上线风险接受

上线前需由业务方和技术方共同确认以下风险是否可接受：

- AI 生成内容存在不稳定性，需要人工审核。
- 外部模型或 Dify 服务不可用会影响生成类功能。
- 当前项目不内建完整企业级监控和灰度发布能力。
- 简历和文档数据需要按企业安全策略管理。
- 如试点范围扩大，需要补充容量评估和安全评估。

## 13. 回滚方案

### 13.1 回滚触发条件

- 上线后无法登录或核心页面不可访问。
- 数据库迁移导致核心数据不可读。
- 招聘主流程或考试主流程完全不可用。
- 权限失效导致普通用户可访问管理接口。
- 外部 AI 配置错误导致核心生成任务全部失败，且无法快速修复。

### 13.2 回滚准备

- 上线前保存上一版本后端包或镜像。
- 上线前保存上一版本前端静态资源。
- 上线前备份数据库。
- 上线前备份上传文件目录。
- 记录本次配置变更。

### 13.3 回滚步骤

1. 通知业务暂停使用系统。
2. 停止新版本后端服务。
3. 恢复上一版本后端包或镜像。
4. 恢复上一版本前端静态资源。
5. 如涉及数据库迁移，按备份恢复数据库或执行反向迁移。
6. 恢复上传文件目录。
7. 恢复上一版本配置。
8. 启动旧版本服务。
9. 执行健康检查和冒烟测试。
10. 通知业务恢复使用。

### 13.4 回滚后处理

- 记录故障时间线。
- 保留新版本日志、请求样例和错误堆栈。
- 分析根因并补充测试用例。
- 修复后重新走验收流程，不直接二次上线。

## 14. 风险与补救

| 风险 | 影响 | 补救措施 |
| --- | --- | --- |
| LLM/Dify 不稳定 | 生成任务失败或超时 | 增加超时提示、重试机制、人工备用流程 |
| 文档解析失败 | 知识库和试卷不可用 | 限制格式、提示转换、保留原文件 |
| AI 输出不准确 | 业务判断偏差 | 明确 AI 辅助定位、人工审核、保留评分理由 |
| 权限配置错误 | 数据泄露或误操作 | 上线前执行权限用例，管理员接口后端强校验 |
| 上传目录未持久化 | 文件丢失 | 使用持久化卷并纳入备份 |
| 默认密钥未替换 | 安全风险 | 上线检查清单强制校验 |
